# Textures — emergent from properties, zero bundled images

> Implemented in Phase 4 (`crates/engine/src/texture.rs` + `shaders/world.wgsl`). This note records
> the approach and the (optional) Creative-Commons path, with licenses verified.

## Approach: procedural, generated from physical properties

Every material texture is **synthesized on the CPU from the material's cited optical properties**
(`albedo`, `color_variance`, `metallic`) — there are **no external image files in the engine**, so
there is **zero licensing exposure**. This also keeps faith with the core thesis: appearance is
*emergent from properties*, not pasted on.

- **High-res + mip-mapped.** Textures are 512² with a full mip chain generated by box-downsampling.
  Mipmapping *is* the "let the client scale it down" mechanism — the GPU samples the right level per
  distance; a client can also simply upload fewer/smaller levels.
- **Seamless.** The noise lattice wraps, so textures tile across terrain without visible seams.
- **Look.** Base albedo modulated by multi-octave value noise (grain/mottle) scaled by
  `color_variance`, plus mineral flecks and — for metals — bright sparkle specks.
- **Applied via triplanar mapping** (world shader), so voxel faces and the sphere are textured
  without UVs. A per-material params buffer (`roughness`, `metallic`) drives a specular highlight —
  the **shine** — with metals getting a tighter, tinted highlight (sparkle).
- **Tested natively** (`texture.rs` tests): correct size + full mip chain, mean color tracks albedo,
  different materials look different, and non-flat variation.

`TEX_SIZE` is a single constant — raise it to 1024²/2048² for more detail at the cost of memory
(19 materials × 512² RGBA ≈ 20 MB in the GPU array).

## Optional: user-supplied real textures (Creative Commons)

The engine ships none, but a **material module** (`docs/06`) could let a user drop in a real photo
texture per material. If they do, only truly free sources should be used. We verified licenses:

| Source | License | Attribution? | PBR maps? | Notes |
|---|---|---|---|---|
| **ambientCG** (ambientcg.com) | **CC0 1.0** | No | Yes (color/normal/rough/metal/AO/height) | Best breadth (rock, basalt, sandstone, soil, gravel, concrete, metal, snow…), multi-res to 16K; bundling explicitly allowed. **Recommended.** |
| **Poly Haven** (polyhaven.com) | **CC0** | No | Yes, photoscanned, 8K+ | Also excellent; smaller material count than ambientCG. |
| 3DTextures.me | CC0 | No | Yes (≤4K) | Secondary. |
| Texture Ninja | CC0 | No | Photos only (not PBR sets) | Good for calibration/albedo reference. |
| Wikimedia Commons | Per-file (filter to CC0/PD) | Depends | Photos only | CC-BY/CC-BY-SA require credit; **CC-BY-SA can conflict with MIT** — filter to CC0/PD only. |
| ShareTextures | "custom CC0" w/ redistribution limits | No | Yes | ⚠️ Restriction is inconsistent with true CC0 — reference only, don't bundle. |

**Guidance:** the engine stays pure-procedural (no images, no attribution burden). If real textures
are ever wanted, prefer **ambientCG** or **Poly Haven** (unambiguous CC0, redistribution allowed),
and treat any non-CC0/PD source as calibration reference only — never bundle it in the MIT repo.

## Possible future work

- Calibrate the procedural parameters against CC0 reference albedo/roughness for extra realism.
- Generate normal maps procedurally (from the height/grain field) for relief lighting.
- Anisotropic grain (wood, bedded sandstone) by stretching the noise along a per-material direction.
